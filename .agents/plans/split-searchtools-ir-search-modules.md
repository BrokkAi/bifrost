# Split searchtools.rs, semantic/ir.rs, and structural/search.rs along their real module boundaries

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds. This document must be maintained in accordance with `.agents/PLANS.md` at the repository root.

## Purpose / Big Picture

Three of the repository's largest source files each contain more than one genuine abstraction cohabiting for historical reasons, and their size is now a tax on every edit (the #1057 work touched two of them). After this change, each file becomes a module directory whose children mirror the boundaries that already exist in the code, with the parent `mod.rs` re-exporting the exact previous public surface so that **no file outside the split module changes**. This is a pure code-motion refactor: no behavior change, no API change, no logic edits. Success is observable as: the same test suites pass before and after, `cargo clippy --all-targets --all-features -- -D warnings` stays clean, and no consumer file appears in the diff.

The three targets and their real seams:

1. `src/searchtools.rs` (9,056 lines) — several independent MCP tool families (scan_usages, symbol sources, summaries, definitions-by-reference, search/navigation) plus a shared selector-resolution core, all in one file with repo-wide fan-in.
2. `src/analyzer/semantic/ir.rs` (6,643 lines) — the semantic IR type contract, a `validate_*` checker family that is an algorithm *over* that contract, and 2.1k lines of inline tests.
3. `src/analyzer/structural/search.rs` (11,001 lines) — the CodeQuery result/diagnostic contract consumed by rendering and policy layers, the query execution engine, and 2.9k lines of inline tests.

## Surprises & Discoveries

- Observation: the receiving side of two splits already follows the target conventions, so the refactor extends existing patterns rather than inventing them.
  Evidence: `src/analyzer/semantic/mod.rs:23` is `pub use ir::*;` (the ir surface is already glob-re-exported, so an `ir/mod.rs` that re-exports its children changes nothing for the ~15 consumer files); `src/analyzer/structural/query/` already contains a dedicated `tests.rs` child module.

- Observation (M1): a handful of "definition candidate" rendering primitives (`DefinitionCandidateRenderCache`, `definition_candidate`/`definition_candidates`/`*_with_cache`, `definition_candidate_from_range`, `definition_candidate_key`, `declaration_kind_name`, `lexical_definition_candidate`, plus `DefinitionCandidate`/`DefinitionCandidateKey`/`DefinitionOutcomeKey`/`DefinitionDiagnostic`) are called from **both** `navigation.rs` (the by-location renderers `render_definition_lookup`/`render_type_lookup`) and `definitions.rs` (the by-reference renderer `render_definition_reference_lookup`/`semantic_outcome_key`) — they are not family-specific despite living textually between the two call sites in the original file. Placed in `selectors.rs` (see Decision Log). Verified precisely by cross-referencing which bucket's generated file textually invoked which other bucket's names (build script + `\b`-boundary regex over a string/comment-aware lexer, to rule out doc-comment mentions), not by guesswork.
- Observation (M1): converting a directly-declared `pub(crate)` item into a `pub(crate) use child::Item;` re-export can newly trip `unused_imports` in a `cargo clippy --all-targets` "lib-only" (non-test) sub-compilation, even though the item compiled warning-free at its original definition site. Rust's `dead_code` lint is exempt for any item with `pub`/`pub(crate)`/`pub(super)` visibility (assumed reachable from outside the current translation unit), but `unused_imports` is not exempt for `pub(crate) use` specifically, since the compiler can fully verify crate-wide reachability. `ScanUsagesSurface` hit this: its only real consumer is the moved `#[cfg(test)] mod tests`, which the "lib" sub-target compiles without `cfg(test)`. Fixed with a scoped, commented `#[allow(unused_imports)]` on that one re-export line rather than dropping the re-export (dropping it would violate the "identical previous surface" contract).
- Observation (M1): struct/enum field privacy does not travel with a blanket item-level `pub(super)` widening. Two structs (`SummaryTargets`, `UsageHitRow`) had private fields that the moved test module constructs via struct-literal syntax from a sibling child module; Rust's field privacy is independent of the struct's own visibility, so each field needed its own `pub(super)` bump (enum variant fields do not have this problem — variant fields always share the enum's own visibility).

- Observation (M2): `ir.rs`'s entire `validate_*`/`measure_artifact_work`/`account_*` family, and every one of their private helper types (`Boundaries`, `ControlEdgeIndex`, `GapIndex`, plus the type aliases `CaptureDestinationIndex`/`ProcedureLocatorIndex`/`AsyncSuspendIndex`), was already file-private (no `pub` or `pub(crate)` anywhere in the family) — so, contrary to the plan text's assumption that a `semantic::ir::validate_*` external path might exist to preserve, there was no such path: the only re-export burden was internal. Verified with a name-list `grep`/Python scan across the whole file for all 48 family names plus the 6 helper-type names, filtered to hits outside the validate line range — the only 3 hits were the 3 cross-calls described below.
- Observation (M2): exactly three symbols cross the contract/validate boundary in the "parent calls into child" direction: `SemanticArtifact::try_new_with_budget` (contract, in `mod.rs`) calls `measure_artifact_work`, `validate_artifact`, and `find_boundaries` (all three defined in what became `validate.rs`). Since Rust privacy only flows to *descendant* modules and `ir::validate` is a *child* of `ir`, `mod.rs` cannot see these without a bump; found by grepping the exact 48+6-name list against the whole file and filtering to hits landing outside the validate/tests line ranges. `find_boundaries` additionally forced `pub(super)` onto its return type `Boundaries` and each of that struct's three fields (the same "field privacy needs its own bump" lesson from M1's `SummaryTargets`/`UsageHitRow`, but discovered here for a *return-type* dependency rather than a test-side struct literal).
- Observation (M2): the reverse direction needed zero changes. Every private item defined in `mod.rs` (types, fields, and one entangled private method, `ControlFlowGraph::validate`) is already visible from both `validate.rs` and `tests.rs` with no edits, because Rust visibility is "defining module + all descendants," and both children are descendants of `ir`. This is the mechanism the plan's Context section already named (point (c)); M2 confirmed it holds symmetrically for a contract/algorithm split, not just for a tests-only split.
- Observation (M2): the moved test module (`tests.rs`) has zero direct references to any `validate_*`/`measure_artifact_work` function or private helper type — confirmed by grepping the extracted test body against the full 48+6-name list with zero hits. Tests exercise validation only indirectly through the public `SemanticArtifact::try_new`/`try_new_with_budget` API, so no additional re-export or `use` plumbing was needed for `tests.rs` beyond what it already had (`use super::*;`, unchanged).

- Observation (M3): `apply_pipeline_step`, the engine's single per-step dispatcher, calls into essentially every helper in the file's "expansion zone" (either directly or transitively), so a naive call-graph reachability check from `apply_pipeline_step` marks almost everything as "engine-reachable" and would leave nothing for `expansions.rs`. The naive check has to be corrected by *not* descending into the three named primitives themselves when computing what the engine reaches by other paths, then comparing that restricted "engine reach" against the full reach of the three named primitives. Concretely: `outbound_reference_expansions` (the per-step dispatch shim for `Uses`/outbound reference steps) is itself called directly by `apply_pipeline_step`, so it stays in `mod.rs` even though it exists only to prepare and call `scan_outbound_reference_hits` — this is exactly why the plan names `scan_outbound_reference_hits`, not `outbound_reference_expansions`, as the primitive that moves. The same pattern holds for `receiver_analysis_expansions`, `direct_member_expansions`, `call_site_expansions`, `call_input_expansions`, and `expand_hierarchy`: all are direct per-step dispatch helpers called from `apply_pipeline_step`, so all stay in `mod.rs`, even though `call_site_expansions`/`call_input_expansions` sit textually beside the call-declaration/inbound-reference clusters that do move.
  Verified with a Python script over the whole file: for every function defined in the 4000–7220 line range, find every bare-word occurrence (not just call-syntax `name(` — see next entry) inside every other such function's body, build a caller→callee graph, then compute reachability from `apply_pipeline_step` (stopping recursion at, but still counting, the three named primitives) versus full reachability from the three named primitives. The set-difference (named-reachable minus engine-reachable) is the exclusive/expansions.rs set; the intersection is the shared/stays-in-`mod.rs` set.

- Observation (M3): a call-graph script that greps for `name\s*\(` (call syntax) silently misses a function referenced as a bare function pointer, e.g. `.min_by_key(primary_range_key)` and `.sort_by_key(primary_range_key)` (no trailing paren at the use site). `primary_range_key` looked like a two-call, expansions-only helper under the call-syntax-only grep, but a bare-word (`\bprimary_range_key\b`, no trailing-paren requirement) grep found two additional bare-fn-pointer uses inside `IndexedDeclarations::get` and `structural_receiver_ranges` — both engine-internal, both outside the expansion zone — which correctly keeps `primary_range_key` in `mod.rs` as a shared helper. Re-ran the full exclusive/shared classification with the bare-word regex and got an identical exclusive/shared partition for every other candidate, so this was a near-miss rather than a systemic error, but it is the reason every call-graph claim in this milestone was cross-checked with a bare-word grep, not just a call-syntax grep.

- Observation (M3): manual line-number arithmetic for small multi-line `struct { ... }` bodies is unreliable at this file size — a manual scan misplaced `CallTraversalWork`'s and `CallPathNode`'s closing braces by one line each (attributing a subsequent blank line, or the next item's `struct` keyword, to the wrong chunk). Recovered by writing a brace-counting `block_end(start_line)` helper (string/char-literal-aware) driven off a regex-anchored signature-line search, then cross-checking its output against every chunk boundary already derived by hand; every other hand-derived boundary matched the automated tool exactly, isolating the two small-struct off-by-one errors before they reached the extraction step. All 21 chunk boundaries used in the final extraction script were verified this way before running it.

- Observation (M3): widening `group_outbound_lookup_candidates` to `pub(super)` (needed because `tests.rs` calls it directly) tripped `private_interfaces` on its still-private return type `OutboundLookupCandidates` (`cargo check --all-targets` fails this as a hard error, not a lint warning, when the type is less visible than the function exposing it). Fixed by widening the struct itself to `pub(super)`; its two `BTreeMap`-typed fields (`by_target`, `sites`) stayed private since only their *names* (`ambiguous_sites`, `ambiguous_candidates_complete`, `omitted_sites`) are read by `tests.rs`'s struct-literal-free field access — the same "field privacy needs its own bump, independent of the struct's own visibility" lesson M1 found for `SummaryTargets`/`UsageHitRow`, confirmed again here for a third split.

## Decision Log

- Decision: hard review contract for all three milestones — the diff must contain **zero changes outside the split module's directory** (plus the deleted original file). Every consumer keeps compiling against the same paths because the new `mod.rs` re-exports the previous public surface verbatim.
  Rationale: this is what makes a 9–11k-line move reviewable at all: the reviewer checks the re-export list and the compile/test gates instead of re-reading moved code. Fan-in is large (`searchtools::` is imported by ~20+ files across src/ and tests/).
  Date/Author: 2026-07-22 / Claude (approved scope by Jonathan: these three files, in this order)

- Decision: pure move only — no renames, no signature changes, no visibility narrowing, no drive-by cleanups. Visibility may only be *widened* where the move requires it (`pub(super)`/`pub(crate)` on items that were file-private and are now referenced across sibling child modules), and each such widening should be the minimum that compiles.
  Rationale: mixing refactor-by-motion with refactor-by-edit destroys the reviewability property above and turns a safe change into a risky one.
  Date/Author: 2026-07-22 / Claude

- Decision: inline `#[cfg(test)] mod tests` blocks move to a child `tests.rs` (`#[cfg(test)] mod tests;`) inside the new directory, not to `tests/`.
  Rationale: these are unit tests exercising private items; a child module keeps `super::*` access working unchanged. `structural/query/tests.rs` is the in-repo precedent.
  Date/Author: 2026-07-22 / Claude
